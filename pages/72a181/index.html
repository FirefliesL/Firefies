SystemService的启动

SystemService是系统进程中运行的服务的基类，基本上所有的系统服务都是继承自SystemService。

SystemService由SystemServiceManager进行管理，一般启动一个SystemService时，
是调用SystemServiceManager的startService(String className)或startService(Class<T> serviceClass),
内部实现是通过反射取到service入参只有Context的单参数构造方法，创建实例，调用实例的onStart()方法。

所以SystemService的启动主要看 onStart() 中的实现。

另外还需要关注一下 onBootPhase, 在SystemServer启动的不同阶段，都会通过SystemServiceManager对已经启动的SystemService进行通知。

目前有一下几个启动阶段：

启动阶段 含义
PHASE_WAIT_FOR_DEFAULT_DISPLAY 系统在启动时发送给系统服务的最早启动阶段
PHASE_LOCK_SETTINGS_READY 在这个启动阶段之后，服务可以获得锁设置数据
PHASE_SYSTEM_SERVICES_READY 在这个启动阶段之后，服务可以安全地调用核心系统服务，如PowerManager或PackageManager
PHASE_DEVICE_SPECIFIC_SERVICES_READY 在这个启动阶段之后，服务可以安全地调用特定于设备的服务
PHASE_ACTIVITY_MANAGER_READY 在这个启动阶段之后，服务可以广播intent。
PHASE_THIRD_PARTY_APPS_CAN_START 在这个启动阶段之后，服务可以启动/绑定到第三方应用程序。应用程序将能够使用Binder进行服务调用
PHASE_BOOT_COMPLETED 在这个启动阶段之后，服务可以允许用户与设备交互。这个阶段发生在启动完成并且主应用程序启动时。为降低延时，系统服务可能更建议监听这个阶段，而不是注册一个广播接收器{@link android.content.Intent#ACTION_LOCKED_BOOT_COMPLETED}

除了PHASE_BOOT_COMPLETED是在AMS中触发的，其他启动阶段都是在SystemServer中触发。

Android中的Makefile

build/make/core/Makefile中看到有 $(hide)的标记，表示引用hide变量，全局搜索”hide :=”的字眼，可以发现build/make/core/config.mk中有

说明所有的$(hide) xxxx语句，其实就等同于命令前加了“@”字符，即在Makefile中表示的时关闭了回显，说明Makefile在执行@后面的命令时，不回显将要执行的命令。

Makefile中，格式为这样的

target : prerequisties
目标：需要依赖的条件

但如果文件多了，按部就班写会显得很长，所有这时候makefile中的常用命令就产生了，如下：

  • $@ 表示目标文件
  • $^ 表示所有的依赖文件
  • $< 表示第一个依赖文件
  • $? 表示比目标还要新的依赖文件列表

aidl文件编译成java

build/make/core/java.mk中

主要参数 看 PRIVATE_AIDL_FLAGS，通过-p 指定了预编译的aidl， -I 指定了aidl搜索目录

  1. aidl使用说明
    aidl –lang={java|cpp|ndk} [options] input_aidl_file。
    在命令行中，aidl不带任何参数，打印使用说明。
  2. –lang={java|cpp|ndk}
    指定生成接口类型。如果不指定，默认生成Java文件。
  3. -o 指定输出目录
    Java：省略-o参数，java文件生成到aidl文件目录下。
    C++/Rust：必选参数。
    为了清除查看效果，指定到单独的目录：-o ./java
  4. -I DIR, –include=DIR
    import搜索路径，指定依赖的aidl文件所在目录。不是C++头文件目录。
    例如：IMyServiceInterface.aidl引用了IMyServiceCallback.aidl，在生成IMyServiceInterface.aidl的时候需要指定-I参数才能正常编译。
    只要是AIDL文件中有依赖，生成任何类型的接口（Java、C++、Rust）都要指定这个参数

记录一些关键的tag

关于tag的描述可以参考 system/core/logcat/event.logtags
格式：(<name>|data type[|data unit])
data type:
1: int
2: long
3: string
4: list
5: float
data unit:
1: Number of objects
2: Number of bytes
3: Number of milliseconds
4: Number of allocations
5: Id
6: Percent
s: Number of seconds (monotonic time)
Default value for data of type int/long is 2 (bytes).

AAOS多用户开启/关闭

多用户的内容可以参考官方文档：
https://source.android.google.cn/docs/devices/admin/multi-user#applying_the_overlay
https://source.android.google.cn/docs/devices/admin/multi-user#enabling-the-headless-system-user
https://source.android.google.cn/docs/automotive/users_accounts/multi_user

在搭载车载feature的AOSP中(AAOP [Android Automotive OS])，默认启用了Headless system user,官方文档有说明如何启用的。

同时更详细的汽车多用户也可以在官方文档中找到.
Headless system user就是启动时初始化的第一个用户，不过这个用户一直运行在后台，需要额外的前台用户才能进行界面交互

根据官方文档所说，开启关闭多用户只需要禁用headless system user即可

  • 启用Headless system user后，需要将config_multiuserMaximumUsers的值设置为2(或以上)，
  • 关闭多用户

ro.fw.mu.headless_system_user属性值控制是否启用无头系统用户
android.car.number_pre_created_users预创建的user
android.car.number_pre_created_guests预创建的guest
android.car.user_hal_enabled是否启用用户HAL属性

如果禁用了多用户，用户HAL属性没有打开的必要，引用官方的说明：

从 Android 11 开始，Android Automotive OS (AAOS) 在 VHAL 上引入了一组新属性，用于创建、切换、删除和关联外部配件以识别用户。例如，这些新属性使驱动程序能够将外部配件（例如密钥卡）与其 Android 用户配对。然后，当驾驶员接近车辆时，ECU 被唤醒并检测到钥匙扣。该 ECU 向 HAL 指示信息娱乐系统应启动哪个 Android 用户，从而减少驱动程序等待其 Android 用户加载的时间。
