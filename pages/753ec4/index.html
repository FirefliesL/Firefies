DreamManagerService

DreamManagerService是处理屏保的系统服务

涉及到的几个类

类名 简要说明
DreamManagerService 屏保服务的实现类
PowerManagerService 和DreamManagerService有部分交互
DreamController 控制屏保服务的具业务操作类
DreamService Service的子类，自定义的屏保需要继承自它
DreamActivity 屏保展示页

DreamManagerService启动流程

DreamManagerService是继承自SystemService，那么一般就是在SystemServer中启动的(SystemService由SystemServiceManager管理，SystemServiceManager是在SystemServer中初始化的）

DreamManagerService是在SystemServer中的startOtherServices中启动的

启动时将IDreamManager的实例对象加入到了ServiceManager，服务名称dreams

触发屏保

  sequenceDiagram

% 调用DreamManagerService的dream方法 %
note over DreamManagerService:dream()
note over DreamManagerService:requestDreamInternal()
DreamManagerService->>PowerManagerService:nap()
PowerManagerService->>DreamManagerService:startDream() 
DreamManagerService->>DreamController:startDream()
DreamController->>DreamService:bindServiceAsUser()

DreamService-->>DreamController:返回DreamServiceWrapper对象
DreamController->>DreamService:attach()
note over DreamService:ActivityTaskManager.getService().startDreamActivity()
  1. 调用DreamManagerService的dream()，进入requestDreamInternal()方法，内部去调用PowerManagerService的nap()方法
  2. 在PowerManagerService的napInternal方法中调用updatePowerStateLocked()更新电源状态，
    一系列逻辑处理后，主要在updateDreamLocked()中发送消息到内部Handler，执行handleSandman()方法，在handleSandman中又调用DreamManagerService的startDream()。
  3. 回到DreamManagerService的startDream()，实际调用了DreamController的startDream()。
  4. 在DreamController中bind了DreamService，在onServiceConnected中调用了DreamService中实现的attach，attach中去ActivityTaskManagerService的startDreamActivity()去启动DreamActivity,并通过intent传递了binder对象
  5. 在DreamActivity的onCreate()方法中取到了DreamService的binder对象(DreamServiceWrapper)，回调DreamService的binder中实现的onActivityCreated，在DreamService中保存了Activity信息

结束屏保

  sequenceDiagram

note over DreamManagerService:awaken()
note over DreamManagerService:requestAwakenInternal()
note over DreamManagerService:stopDreamLocked(false /*immediate*/, "request awaken")
DreamManagerService->>DreamController:stopDream()
note over DreamManagerService,DreamController:immediate = false
DreamController->>DreamService:wakeUp()
note over DreamService:onWakeUp()
DreamService-->>DreamService:finish()
note over DreamService:finish()中调用activity.finishAndRemoveTask()，关闭屏保页
DreamService->>DreamManagerService:finishSelf()
note over DreamManagerService:stopDreamLocked(true /*immediate*/, "finished self")
DreamManagerService->>DreamController:stopDream()
note over DreamManagerService,DreamController:immediate = true
DreamController->>DreamService:detach()
DreamController->>DreamService:unbindService()
  1. 调用DreamManagerService的awaken()，进入requestAwakenInternal()方法，最终调用的stopDreamLocked(false /immediate/, “request awaken”)方法
  2. stopDreamLocked()中去调用DreamController的stopDream()方法
  3. 当stopDream()入参的immediate为false时，调用的是DreamService中的wakeUp()方法，触发onWakeUp()->finish()
  4. 在finish()中取到之前保存的DreamActivity,调用activity.finishAndRemoveTask()，关闭屏保页
  5. 然后调用DreamManagerService的finishSelf(mDreamToken, true /immediate/)，会再走一次stopDreamLocked()，
    去清除DreamManagerService中保存的相关数据，然后进入DreamController的stopDream()方法
  6. 当stopDream()入参的immediate为true时，调用的是DreamService中的detach()
  7. 最后DreamController执行unbind DreamService。

注：DreamActivity的ActivityType是ACTIVITY_TYPE_DREAM

isValidLaunchStack

记录一次屏保无法退出的现象

现象

进入屏保页后，点击屏保无响应，无法退出，无法回到桌面，通过adb直接下发指令启动Activity也无法成功（stack 不在栈顶，在栈顶下面一层）

原因

后面发现DreamActivity的ActivityType是ACTIVITY_TYPE_DREAM，普通APP的ActivityType是ACTIVITY_TYPE_STANDARD，
由于要启动的Activity的栈和顶层栈的类型不同，所以无法打开。

但是DreamService中实现了Window.Callback，并将DreamActivity的Window.Callback设置成了DreamService，
而DreamService实现的Window.Callback,对dispatchTouchEvent做了处理，会去调用wakeUp()退出屏保，所以正常情况下，触摸屏幕就会退出屏保。

最后发现有其他进程在打开屏保时，刚好去唤醒屏幕，将DreamService中的mFinished标志置为了true，
后续屏保页成功显示后，由于mFinished已经置为true，执行解锁时无法进入解锁逻辑。屏保一直无法退出。

解决方法

DreamService中添加字段mActivityCreated记录屏保页是否成功启动，未启动前不执行解锁操作
