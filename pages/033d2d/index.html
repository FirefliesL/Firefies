分析Android11开始无法后台弹出Toast的原因

toast调用的就是NotificationManagerService中的enqueueToast

其中有限制了自定义toast的弹窗，判断当前的pkg是否被block，如果是block就不让弹出

接着看一下CompatConfig这个类，发现这个类在构造方法中去读取了指定目录下的xml配置文件，将xml解析成 CompatConfig。而这个CompactConfig中就包含了所有的CompatChange。有一个疑问点，这个CompatChange只看到解析，没有看到在那里生成的。

现在知道，Toast时，会去判断限制后台自定义toast这个新特性（CompatChange）是否启用，判断的逻辑是读取xml配置文件，从文件中解析出CompatChange列表，但目前没有看到这个配置文件是如何生成的。

去看platform-compat-config.xsd，注释里又写着这是个模板，真正生成xml的是com.android.compat.annotation.ChangeIdProcessor,所以配置文件是在编译时生成的，是根据注解来生成的配置文件

接下来，就看最关键的一步，CompatChange是如何判断指定app是否启用该特性的：

后台弹出Toast的方法

所以在Toast之前，

  1. 先去判断限制后台自定义toast新特性在当前app中是否重写过启用状态，重写过就用重写的状态，
  2. 没有重写，判断是否禁用，禁用就直接返回
  3. 没有禁用，判断app的targetSdk是否大于注解中指定的sdk，大于则启用
  4. 以上条件都不满足，默认启用

想要让app后台弹toast就有几种方式：

  1. 降低app的targetSdk，不能大于29
  2. 修改 NotificationManagerService中CHANGE_BACKGROUND_CUSTOM_TOAST_BLOCK的注解，将开始启用的sdk值改得比app的targetSdk更大
  3. NotificationManagerService的CHANGE_BACKGROUND_CUSTOM_TOAST_BLOCK添加@Disabled注解，禁用该特性
  4. 重写指定app的启用状态

目前看来重写指定app的启用状态，影响范围最小，重写启用状态，需要调用CompatChange的addPackageOverride方法，往上找，看有没有什么api，最后发现在PlatformCompat中有setOverrides(CompatibilityChangeConfig overrides, String packageName)来重新设置特性启用状态，但是传入的参数是多个特性的复合体，使用CompatConfig的addOverride控制更精确，可以在PlatformCompat中对CompatConfig构造完成时，修改一下指定app的启用状态
